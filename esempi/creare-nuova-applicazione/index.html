    Creare una nuova applicazione¶

    Preparazione¶

    Per prima cosa conviene recarsi nel backend all'indirizzo /backend/settings/cache e controllare che la modalità sviluppatore sia attiva e tutti i sistemi di cache siano disattivati.

    Supponiamo di avere già un template frontend custom ( Vedi esempio Creare un nuovo template frontend) di nome "primo".

    Applicazioni del template¶

    Le applicazioni di Echidna possono essere dichiarate in 2 posizioni diverse:

    • dentro la cartella /app/applications ci sono le applicazioni "core", condivise con tutti i template
    • dentro la cartella del template frontend in applications ( es : /app/frontend/primo/applications) che saranno disponibili solo quando il template è in uso

    E' buona norma creare le proprie applicazioni personalizzate direttamente dentro al template frontend.

    Creare l'applicazione "Prodotti"¶

    Innanzitutto creiamo la cartella applications all'interno del nostro template frontend.

    Dentro questa cartella creiamo una directory denominato "prodotti" che conterrà la nostra applicazione ed i file necessari.

    path template frontend
                | applications
                    | prodotti
                        | entities
                            | Prodotto.php
                        | ProdottiApplication.php
                        | ProdottiFrontend.php
                        | ProdottiBackend.php

    ProdottiApplication.php¶

    Conterrà la classe madre della nostra applicazione Prodotti.

    L'applicazione madre deve estendere core\abstracts\Application e implementare i 3 metodi astratti.

    <?php 
namespace frontend\primo\applications\prodotti;

use core\abstracts\Application;
use frontend\primo\applications\prodotti\entities\Prodotto;

class ProdottiApplication extends Application{
    static function getBackendApplication()
    {
        return ProdottiBackend::class;   
    }
    static function getFrontendApplication()
    {
        return ProdottiFrontend::class;
    }
    static function getEntityClass()
    {
        return Prodotto::class;
    }
}

    L'applicazione madre si occupa di dichiarare quale entità gestisce e quali sono le sotto applicazioni a cui delegare la gestione del backend e del frontend.

    ProdottiBackend.php¶

    L'applicazione che gestirà il backend deve estendere la classe core\abstracts\BackendApplication e la sua minima implementazione è questa:

    <?php 
namespace frontend\primo\applications\prodotti;

use core\abstracts\BackendApplication;

class ProdottiBackend extends BackendApplication{
    static function getApplication()
    {
        return ProdottiApplication::class;
    }    
}

    In questa forma l'applicazione ProdottiBackend crea già le route di base per gestire l'entità "Prodotto" come liste, maschere di aggiunta e modifica.

    Possiamo aggiungere, rimuovere o modificare le route effettuando l'override del metodo declareRoutes().

    In questo esempio creiamo una nuova route accessibile tramite /backend/prodotti/routeprova che viene gestita dal metodo "routeProva" della nostra classe backend

        static function declareRoutes()
    {
        $routes = parent::declareRoutes();
        $routes["prodotti.backend.routeprova"]  = new Route("","routeprova",[static::class,"routeProva"]);
        return $routes;
    }
    static function routeProva( $params = [] ){
            echo "Questa è una route di prova";
            exit;
    }

    Attivare la nostra applicazione¶

    Per attivare la nostra applicazione dobbiamo creare la cartella config dentro il nostro tema con all'interno il file applications.php.

    Il file applications all'interno delle configurazioni del template ha la stessa funzione del file applications.php situato in /app/config, quella di dichiarare al applicazioni da utilizzare.

    La differenza è che il file interno al template verrà utilizzato solo nel caso in cui il tema sia attivo e andrà ad integrare il file applications globale, aggiungedo o sovrascrivendo le applicazioni da utilizzare.

    Contenuto del file applications.php interno al template frontend per il nostro esempio

    <?php
use frontend\primo\applications\prodotti\ProdottiApplication;

return [
    "prodotti"  =>  ProdottiApplication::class
];

    ProdottiFrontend.php¶

    Questa è la classe che gestisce la parte frontend della nostra applicazione. Può anche non essere dichiarata ( in questo caso il metodo getFrontendApplication() della nostra applicazione dovrà ritornare null) nel caso in cui l'applicazione non debba gestire route lato frontend.

    Anche l'applicazione frontend può dichiarare le route da gestire esattamente come l'applicazione backend facendo l'override del metodo declareRoutes().

    Creiamo una route "pagina prodotto" : 

    static function declareRoutes()
    {
        $routes = parent::declareRoutes();

        $routes["prodotti.frontend.paginaprodotto"] = new Route("","/paginaprodotto",[static::class,"paginaProdotto"]);

        return $routes;
    }

    e il metodo che dovrà gestire la richiesta

    static function paginaProdotto( $params =[] ){
        return [
            "prodotti/pagina-prodotto",[
                "variabilediprova" => "Io sono una variabile"
            ]
        ];
    }

    Il metodo paginaProdotto ritorna un array di 2 elementi, che permette al template engine di effettuare il rendering della risorsa richiesta.

    Il primo valore dell'array rappresenta il path al template richiesta, partendo dalla route dei nostri template.

    Il secondo elemento è un array chiave/valore con i dati da passare al template.

    In questo caso stiamo dicendo di renderizzare il template prodotti/pagina-prodotto che rappresenta il file /app/frontend/primo/template/prodotti/pagina-prodotto.twig. 

    Questa è la nostra pagina prodotto
<br>
e questa è la variabile : {{variabilediprova}}

    Entità e come modellare il database¶

    Abbiamo dichiarato che l'applicazione Prodotti dovrà utilizzare come entità principale "Prodotto", che si trova fisicamente in /app/frontend/primo/applications/prodotti/entities/Prodotto.php

    Nota

    Normalmente la tabella generate equivale al nome della classe minuscolo ( nel nostro caso "prodotto" ) ma può essere impostata sovrascrivendo static function getTable() e ritornando il nuovo nome

    static function getTable(){
    return "nometabellaprodotto";
}

    Per costruire la struttura della tabella possiamo fare l'override del metodo static function schema() e ritornare un array di oggetti Field.

    In questo esempio la nostra tabella prodotto avrà questa struttura

    • id -> campo prima autoindex
    • nome -> campo varchar lunghezza 256
    • prezzo -> campo float con precisione 2 decimali
    • url -> la url relativa alla quale sarà disponibile la risorsa
    • immagine -> campo varchar lunghezza 512 contenente la url dell'immagine

    Stiamo inoltre dichiarando che i campi nome,url, prezzo e immagine sono editabili da backend e che il campo immagine dovrà utilizzare, per i form, il template media ( presente in backend/template/fields) che richiama l'editor di immagini dell'applicazione media.

    <?php 
namespace frontend\primo\applications\prodotti\entities;

use core\db\Field;
use core\Model;

class Prodotto extends Model{
    static function schema()
    {
        return [
            "id"        =>  Field::primaryIndex(),
            "nome"      =>  Field::varchar(256)->editable(),
            "prezzo"    =>  Field::float(10,3)->editable(),
            "url"       =>  Field::varchar(256)->editable(),
            "immagine"  =>  Field::varchar(512)->editable()->setTemplate("media")
        ];
    }
}

    Creare menù backend¶

    Adesso abbiamo l'entità pronta e l'applicazione backend già disponibile a gestire le route di modifica base. Ad esempio la lista dei nostri prodotti sarà disponibile qua : /backend/prodotti/lista

    Non abbiamo però inserito la nostra applicazione nel menù del backend.

    Possiamo farlo con il seguente codice nella classe ProdottiApplication, facendo l'override del metodo init.

    use delle classi

    use backend\BackendTemplate;
use backend\Menu;
use core\abstracts\Application;
use core\services\RouterService;

    metodo init sovrascritto

    static function init( $n ){
        parent::init($n);

        $menu = new Menu("Prodotti",ProdottiBackend::getIcon());
        $menu->addItem("Lista prodotti", RouterService::getRoute(Prodotto::getEntity().".list"));
        BackendTemplate::addMenu($menu);
    }

    Collegare un prodotto lato frontend¶

    Dopo aver creato un prodotto tramite il backend, possiamo utilizzare i metodi di ricerca dei Model per caricare i dati dal database e passarli dal template.

    Il metodo paginaProdotto all'interno di ProdottiFrontend può essere modificato così

    static function paginaProdotto( $params =[] ){
        $prodotto = Prodotto::findById(1);

        return [
            "prodotti/pagina-prodotto",[
                "prodotto" => $prodotto
            ]
        ];
    }

    Per recuperare il prodotto con id 1 e passarlo al template.

    All'interno del template possiamo poi accedere al prodotto tramite la variabile "prodotto" che sarà una istanza della nostra entità Prodotto.

    Modifichiamo il template in questo modo per visualizzare le informazioni del prodotto.

    {% extends "base.twig" %}

{% block content %} 
    Questa è la nostra pagina prodotto
    <br>
    <p>
        Nome prodotto : {{prodotto.nome}}
        <br>
        Prezzo : {{prodotto.prezzo | price}}
        <img src="{{prodotto.immagine}}" />
    </p>
{% endblock %}

    Il filtro twig "price" si occupa di formattare un numero come prezzo in €
